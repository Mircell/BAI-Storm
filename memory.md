# مستندات فنی پروژه BAI-Storm

این فایل به‌عنوان حافظه (Memory) فنی پروژه BAI-Storm نگهداری می‌شود و شامل جزئیات معماری، ساختار کد، نقاط ورود API، کلاس‌های اصلی و نحوه‌ی کارکرد سیستم است. برای توسعه‌دهندگان و برای حفظ یکپارچگی دانش پروژه ضروری است.

---

## ۱. نمای کلی پروژه

BAI-Storm یک ابزار تحت وب است که با استفاده از چندین عامل هوش مصنوعی (Agent) جلسات طوفان فکری برگزار می‌کند. این ابزار دو حالت اصلی دارد:

- **طوفان فکری استاندارد (Standard Brainstorming)**: تعداد دلخواه ایجنت (۱ تا ۲۰) به نوبت ایده‌هایی در مورد یک مسئله ارائه می‌دهند. هر ایجنت می‌تواند شخصیت خاص خود را داشته باشد.
- **تکنیک شش کلاه تفکر (Six Thinking Hats)**: شش ایجنت با کلاه‌های سفید، قرمز، سیاه، زرد، سبز و آبی به ترتیب دیدگاه‌های مختلفی را در مورد یک ایده/مسئله ارائه می‌دهند.

پروژه با استفاده از فریم‌ورک Flask ساخته شده و از API‌های سازگار با OpenAI (از جمله پروکسی‌های محلی مانند Ollama) برای تولید محتوای ایده‌ها استفاده می‌کند. جلسات به صورت موقت در حافظه (In-Memory) نگهداری می‌شوند.

## ۴. کلاس‌های اصلی

همه کلاس‌های جلسه از کلاس پایه انتزاعی `BaseSession` (تعریف شده در `models/base.py`) ارث‌بری می‌کنند که متدهای زیر را الزامی می‌کند:

- `get_next_idea()`
- `stop()`
- `get_summary()`

### ۴.۱. کلاس `BrainstormSession` (`models/brainstorm.py`)

**مسئولیت:** مدیریت جلسات طوفان فکری استاندارد.

#### متدها و ویژگی‌ها:

- **`__init__(self, num_agents, problem, base_url, api_key, model, personalities=None, agent_names=None)`**
  - `num_agents`: تعداد ایجنت‌ها (۱ تا ۲۰)
  - `agent_names`: (اختیاری) لیست نام‌های دلخواه برای هر ایجنت
  - `problem`: مسئله یا موضوع
  - `base_url`: آدرس API مدل
  - `api_key`: کلید API
  - `model`: نام مدل (مثل `thinking_not_search`, `llama2`, `gpt-3.5-turbo`)
  - `personalities`: لیستی از شخصیت‌ها (اختیاری) به طول `num_agents`

- **`get_next_idea()`**
  - ایجنت بعدی را فعال می‌کند، درخواست به API ارسال می‌شود و ایده‌ی جدید دریافت می‌گردد.
  - بازگشت: `(agent_name, idea)`
  - در صورت پایان جلسه یا توقف، `(None, None)` برمی‌گرداند.

- **`stop()`**: جلسه را متوقف می‌کند.

- **`get_summary()`**: خلاصه‌ای از جلسه شامل همه‌ی ایده‌ها و وضعیت برمی‌گرداند.

#### منطق تولید ایده:

- برای هر ایجنت، سیستِم‌پرامپت از `personality` (شخصیت) گرفته می‌شود.
- پرامپت کاربر شامل `problem` و ایده‌های قبلی (به ترتیب) است.
- درخواست به `{base_url}/chat/completions` با متد POST ارسال می‌شود.
- پاسخ به‌عنوان ایده ذخیره می‌شود.

---

### ۴.۲. کلاس `SixHatsSession` (`models/six_hats.py`)

**مسئولیت:** مدیریت جلسات تکنیک شش کلاه تفکر.

#### متدها و ویژگی‌ها:

- **`__init__(self, problem, base_url, api_key, model)`**
  - پارامترها مشابه `BrainstormSession` اما به جای `num_agents` از شش کلاه ثابت استفاده می‌کند.

- **`get_next_idea()`** (مشابه)
  - ایجنت بعدی (کلاه بعدی) فعال می‌شود.
  - هر کلاه دارای شخصیت از پیش‌تعریف‌شده است (لیست `HATS`).
  - پس از طی شدن همه‌ی کلاه‌ها، چرخه تکرار می‌شود (بدون توقف خودکار).

- **`stop()`** و **`get_summary()`** مشابه کلاس قبلی.

#### لیست کلاه‌ها:

```python
HATS = [
    {'name': 'کلاه سفید (White Hat)', 'color': '#f0f0f0', 'emoji': '⚪', 
     'personality': '... data analyst ...'},
    {'name': 'کلاه قرمز (Red Hat)', 'color': '#e74c3c', 'emoji': '🔴', 
     'personality': '... emotional thinker ...'},
    # ... و بقیه کلاه‌ها
]
```

هر کلاه یک `personality` مخصوص دارد که به‌عنوان سیستِم‌پرامپت به مدل ارسال می‌شود.

### ۴.۳. فایل `models/session.py` (رابط عمومی)

این فایل تنها به عنوان نقطه ورود (Interface) عمل می‌کند و کلاس‌های `BaseSession`, `BrainstormSession` و `SixHatsSession` را از فایل‌های مربوطه import و export می‌کند. به این ترتیب سایر ماژول‌ها (مانند `routes/main.py`) می‌توانند بدون نیاز به دانستن جزئیات فایل‌های جداگانه، از `models.session` کلاس‌ها را دریافت کنند.

## ۵. نقاط API (Routes)

| مسیر | متد | توضیح |
|------|-----|-------|
| `/` | GET | صفحه اصلی (تنظیمات API + لیست ابزارها) |
| `/brainstorm` | GET | صفحه‌ی طوفان فکری استاندارد |
| `/six-hats` | GET | صفحه‌ی تکنیک شش کلاه تفکر |
| `/start` | POST | شروع جلسه‌ی استاندارد. بدنه: `{num_agents, problem, base_url, api_key, model, personalities}` <br> بازگشت: `{session_id}` |
| `/start_six_hats` | POST | شروع جلسه‌ی شش کلاه. بدنه: `{problem, base_url, api_key, model}` <br> بازگشت: `{session_id}` |
| `/next/<session_id>` | GET | دریافت ایده‌ی بعدی از جلسه. بازگشت: `{status, agent, idea, summary, color?, emoji?}` |
| `/stop/<session_id>` | POST | توقف جلسه. بازگشت: `{status: 'stopped', summary}` |
| `/export/<session_id>` | GET | دریافت خروجی HTML خلاصه‌ی جلسه |

**نکته:** تمامی جلسات در یک دیکشنری `sessions` در حافظه‌ی سرور ذخیره می‌شوند (مقادیر موقت).

## ۶. تنظیمات و متغیرهای محیطی

برنامه از تنظیمات زیر در فرم ورودی کاربر دریافت می‌کند:

- `base_url`: آدرس API (پیش‌فرض `http://127.0.0.1:8000/v1`)
- `api_key`: کلید API (پیش‌فرض `sk-local`)
- `model`: نام مدل (پیش‌فرض `thinking_not_search`)

هیچ متغیر محیطی خاصی برای برنامه تعریف نشده است؛ همه‌ی تنظیمات از طریق فرم ارسال می‌شوند.

---

## ۷. نحوه‌ی کار با API مدل

هر دو کلاس از یک منطق مشابه برای ارتباط با API استفاده می‌کنند:

1. ساخت `url = f"{self.base_url}/chat/completions"`
2. ساخت `headers` شامل `Content-Type: application/json` و `Authorization: Bearer {self.api_key}`
3. ساختن `data` با `model`, `messages` (شامل system و user)، `temperature=0.8`, `max_tokens=500`
4. ارسال درخواست `POST` با timeout=60 ثانیه
5. استخراج `idea` از `response.json()['choices'][0]['message']['content']`
6. در صورت بروز خطا، یک پیام خطا به‌عنوان ایده برگردانده می‌شود.

---

## ۸. مدیریت خطا و لاگ‌گیری

- لاگ‌گیری با استفاده از ماژول `logging` با سطح `DEBUG` انجام می‌شود.
- لاگ‌ها هم در فایل `app.log` و هم در خروجی استاندارد (کنسول) نوشته می‌شوند.
- یک `@app.errorhandler(Exception)` عمومی وجود دارد که استثناهای مدیریت‌نشده را به‌صورت JSON با کد ۵۰۰ برمی‌گرداند.

---

## ۹. جلسات و ذخیره‌سازی

- جلسات در دیکشنری `sessions` با کلید `session_id` (UUID) نگهداری می‌شوند.
- هر جلسه یک شیء از کلاس `BrainstormSession` یا `SixHatsSession` است.
- پس از توقف یا پایان جلسه، داده‌ها همچنان در حافظه باقی می‌مانند تا کاربر بتواند خروجی HTML را دریافت کند.
- جلسات به‌صورت دائمی ذخیره نمی‌شوند و با ری‌استارت سرور از بین می‌روند.

---

## ۱۰. خروجی HTML

مسیر `/export/<session_id>` یک صفحه‌ی HTML کامل از خلاصه‌ی جلسه تولید می‌کند. بسته به نوع جلسه، از یکی از دو قالب `export.html` یا `export_six_hats.html` استفاده می‌شود. این قالب‌ها شامل:

- لیست ایده‌ها به ترتیب با نام ایجنت یا کلاه مربوطه.
- **شماره دور** (نوبت) برای هر ایده (در حالت استاندارد و شش کلاه) تا مشخص شود نظر مربوط به کدام دور است.
- **فیلتر بر اساس ایجنت یا کلاه**: کاربر می‌تواند با کلیک روی دکمه‌های فیلتر، فقط ایده‌های یک ایجنت خاص یا یک کلاه خاص را مشاهده کند.
- در حالت شش کلاه، رنگ و ایموجی هر کلاه نیز نمایش داده می‌شود.

این قابلیت‌ها به کاربر کمک می‌کند تا گزارش را به صورت هدفمند بررسی کند و سیر تکامل ایده‌ها را در طول دورهای مختلف دنبال نماید.

## ۱۱. نکات امنیتی

- `api_key` از طریق فرم ارسال می‌شود و در سرور ذخیره نمی‌شود (فقط در حافظه‌ی جلسه نگهداری می‌شود).
- از `os.urandom(24)` برای `secret_key` Flask استفاده شده است.
- ورودی‌های کاربر از طریق فرم اعتبارسنجی می‌شوند (مثلاً عدم وجود `problem` یا `api_key`).
